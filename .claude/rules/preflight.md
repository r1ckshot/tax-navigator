# Preflight (перед стартом задачі)

Передумови закриваються **до** першого рядка коду. Натикатись на них посеред
роботи — означає переробляти або здавати недоперевірене.

## Що перевірити

- **Тести.** Чи є фреймворк і чи він реально запускається тут (`npm test`). Якщо
  задача потребує нового типу тестів (UI, e2e) — перевірити, що він стартує в
  цьому середовищі, ДО того як писати такі тести.
- **Домени.** Якщо задача спирається на зовнішні джерела — звірити allowlist
  (`.devcontainer/init-firewall.sh` + `.claude/settings.json`) і **перевірити
  фактичну доступність**, а не припустити:
  `curl -s -o /dev/null -w '%{http_code}' --max-time 8 https://<домен>/`
- **Команди верифікації.** Чи справді працюють `npm test`, `npm run verify`,
  `npm run build` у цьому середовищі.

## Що робити з результатом

- Передумову, яку можу закрити сам (поставити пакет, додати скрипт) — закриваю
  мовчки і йду далі.
- Передумову, яку може закрити лише Mike (`.claude/settings.json`, firewall,
  доступи) — кажу **відразу, до початку роботи**, і даю готовий блок для вставки.
  Не починаю задачу, мовчки сподіваючись, що обійдеться.
- Сталу межу середовища — дописую в [environment-limits.md](environment-limits.md),
  щоб наступна сесія не відкривала її заново.

## Антипатерн

Спланувати роботу на джерелі/команді, не перевіривши доступність, дійти до
середини й виявити блокер. Одна команда на початку економить перепланування.
