# DECISIONS — журнал рішень (дописується, не переписується)

## 2026-07 (закладено при старті, з чату-стратегії)
- **Продукт: податковий навігатор UA↔PL для IT/B2B.** Чому: 3-етапна валідація (desk → поле R1 → поле R2: 25 органічних питань при порозі 15); рів = доменна matrix + прямий вхід в аудиторію. Відкинуто: CUKR-навігатор (безкоштовний інкамбент Dopomo + дедлайн ринку 03.2027), Shopify-апки (холодний старт проти тисяч), консульські черги (юридично ворожа зона), студентський сегмент (нема податкового болю).
- **Scope: діапазони + порівняння + чекліст, НЕ точні суми і НЕ рекомендації.** Чому: -80% юридичного ризику при -10% цінності; точну арифметику вже роблять inFakt/держкалькулятори — наша цінність у шарі рішення.
- **Юрист: НІ** (бінарно; умови й тригер — OPEN-RISKS §1).
- **Оплата: działalność nierejestrowana → JDG при ~ліміті** (OPEN-RISKS §4). Чому: нуль формальностей на валідаційній фазі, право на JDG за PESEL UKR підтверджене.
- **Верифікація движка: державні калькулятори як еталон** (10/10 профілів) + сценарії Objaśnień MF по резидентству. Чому: сильніший ground truth, ніж думка однієї людини; автоматизується в CI.
- **Воркфлоу: один інструмент** (Claude Code у VS Code), сесія-на-задачу, Plan Mode для планування, стан у STATE/DECISIONS. Чому: Cowork↔Code луп створював ручне носіння повідомлень і другий несинхронний контекст.
- **Технічний фундамент — з матеріалів курсу, не самопис** (starters, sdlc/, workflows, evals). Чому: офіційне, перевірене, збігається з capstone-вимогами.
- **Сім'я (змішане громадянство) = профіль P2 у rules-as-data архітектурі**, після валідації P1 (OPEN-RISKS §6).
- **Ворота G1 перекалібровано: 100 email АБО ≥5 "коли можна купити"** (замість 300). Чому: холодний старт без аудиторії; 300 давало ризик хибно-негативного сигналу.
- **Таймінг: запуск вересень–жовтень 2026** (фініш курсу ~09.09), пік монетизації PIT-сезон лютий–квітень 2027. Домен + Search Console купуються лише на фазі запуску.
- **TG-бот = можливий другий інтерфейс до того ж движка ПІСЛЯ MVP**, не частина ядра. Чому: рішення "ФОП чи JDG" вимагає візуального порівняння; бот — лід-магніт у середовищі аудиторії пізніше.

## 2026-07-16 (M3 capstone, ініціалізація репо)
- **README.md — тільки публічна презентація продукту; операційні правила — окремий CLAUDE.md.** Чому: репо йде в публічний доступ, README читають сторонні (портфоліо), а не тільки Claude Code-сесії; CLAUDE.md автозавантажується кожної сесії, README не мусить дублювати цю роль.
- **docs/notes/, docs/features/, research/ — поза git, лише локально.** Чому: repo публічне; research/tg-mining містить сирі й напівсирі дані реальних людей з приватних/напівприватних Telegram-груп (data-round2/, DATA-COMBINED*.md), а не тільки заявлений `data/`. EVIDENCE.md §3 очищено від дослівних цитат (перенесені в research/tg-mining/EVIDENCE-QUOTES.md, локально).
- **`permissions.deny` для .env: точні suffix-и (`.env.local`, `.env.production.local` тощо), НЕ wildcard `.env.*`.** Чому: `Edit(.env.*)`/`Read(.env.*)` як у starter-базі блокує і `.env.example` (glob матчить будь-що після крапки) — файл-плейсхолдер, який мусить бути редагованим. Разом з цим: `.claude/settings.json` завантажується один раз на старті сесії, правка permissions не підхоплюється "на льоту" — новий деny-патерн діє з наступної сесії, не поточної.
- **4 баги в starter-шаблоні курсу знайдено й виправлено через реальний Docker-runtime тест** (не наші помилки — успадковано зі starter, виявлено лише після повного `make verify` у контейнері):
  1. `init-firewall.sh`: `getent ahostsv6` повертає non-zero коли в домену нема AAAA-запису (норма, не помилка) — з `pipefail` це тихо вбиває весь скрипт на першому такому домені. Фікс: `|| true` на обидва `ips4`/`ips6` присвоєння.
  2. `init-firewall.sh` + `tests/firewall.test.sh`: `curl -fsS` трактує HTTP 404 як "заблоковано", а `api.anthropic.com` на `/` legitimately віддає 404 (API-сервер, немає контенту на root) — фаєрвол насправді пропускав запит. Фікс: прибрано `-f`, залишено лише перевірку на зв'язність (timeout/refused = заблоковано, будь-яка HTTP-відповідь = пройшло).
  3. `tests/firewall.test.sh`: precondition-перевірка стану через `iptables -L OUTPUT` без sudo — non-root `node` user (наш навмисний least-privilege sudoers, лише `init-firewall.sh` passwordless) не має прав це прочитати, тому тест завжди SKIP навіть коли firewall реально активний. Прибрано перевірку, лишились самі curl-тести (не потребують root).
  4. `Dockerfile`: `FROM node:20`, а актуальний `@anthropic-ai/claude-code` вимагає Node ≥22 (EBADENGINE warning при білді). Піднято на `node:22`.
  Після фіксів: `make verify-devcontainer`, `verify-whitelist`, `verify-sandbox`, `verify-firewall` — усі 4 пройшли реально всередині запущеного devcontainer (Docker Desktop, WSL2-бекенд).
