Відкинуто: CUKR-навігатор (безкоштовний інкамбент Dopomo + дедлайн ринку 03.2027), Shopify-апки (холодний старт проти тисяч), консульські черги (юридично ворожа зона), студентський сегмент (нема податкового болю).
- **Scope: діапазони + порівняння + чекліст, НЕ точні суми і НЕ рекомендації.** Чому: -80% юридичного ризику при -10% цінності; точну арифметику вже роблять inFakt/держкалькулятори — наша цінність у шарі рішення.
- **Юрист: НІ** (бінарно; умови й тригер — OPEN-RISKS §1).
- **Оплата: działalność nierejestrowana → JDG при ~ліміті** (OPEN-RISKS §4). Чому: нуль формальностей на валідаційній фазі, право на JDG за PESEL UKR підтверджене.
- **Верифікація движка: державні калькулятори як еталон** (10/10 профілів) + сценарії Objaśnień MF по резидентству. Чому: сильніший ground truth, ніж думка однієї людини; автоматизується в CI.
- **Воркфлоу: один інструмент** (Claude Code у VS Code), сесія-на-задачу, Plan Mode для планування, стан у STATE/DECISIONS. Чому: Cowork↔Code луп створював ручне носіння повідомлень і другий несинхронний контекст.
- **Технічний фундамент — з матеріалів курсу, не самопис** (starters, sdlc/, workflows, evals). Чому: офіційне, перевірене, збігається з capstone-вимогами.
- **Сім'я (змішане громадянство) = профіль P2 у rules-as-data архітектурі**, після валідації P1 (OPEN-RISKS §6).
- **Ворота G1 перекалібровано: 100 email АБО ≥5 "коли можна купити"** (замість 300). Чому: холодний старт без аудиторії; 300 давало ризик хибно-негативного сигналу.
- **Таймінг: запуск вересень–жовтень 2026** (фініш курсу ~09.09), пік монетизації PIT-сезон лютий–квітень 2027. Домен + Search Console купуються лише на фазі запуску.
- **TG-бот = можливий другий інтерфейс до того ж движка ПІСЛЯ MVP**, не частина ядра. Чому: рішення "ФОП чи JDG" вимагає візуального порівняння; бот — лід-магніт у середовищі аудиторії пізніше.

EVIDENCE.md §3 очищено від дослівних цитат (перенесені в research/tg-mining/EVIDENCE-QUOTES.md, локально).
- **`permissions.deny` для .env: точні suffix-и (`.env.local`, `.env.production.local` тощо), НЕ wildcard `.env.*`.** Чому: `Edit(.env.*)`/`Read(.env.*)` як у starter-базі блокує і `.env.example` (glob матчить будь-що після крапки) — файл-плейсхолдер, який мусить бути редагованим. Разом з цим: `.claude/settings.json` завантажується один раз на старті сесії, правка permissions не підхоплюється "на льоту" — новий деny-патерн діє з наступної сесії, не поточної.
- **4 баги в starter-шаблоні курсу знайдено й виправлено через реальний Docker-runtime тест** (не наші помилки — успадковано зі starter, виявлено лише після повного `make verify` у контейнері):
  1. `init-firewall.sh`: `getent ahostsv6` повертає non-zero коли в домену нема AAAA-запису (норма, не помилка) — з `pipefail` це тихо вбиває весь скрипт на першому такому домені. Фікс: `|| true` на обидва `ips4`/`ips6` присвоєння.
  2. `init-firewall.sh` + `tests/firewall.test.sh`: `curl -fsS` трактує HTTP 404 як "заблоковано", а `api.anthropic.com` на `/` legitimately віддає 404 (API-сервер, немає контенту на root) — фаєрвол насправді пропускав запит. Фікс: прибрано `-f`, залишено лише перевірку на зв'язність (timeout/refused = заблоковано, будь-яка HTTP-відповідь = пройшло).
  3. `tests/firewall.test.sh`: precondition-перевірка стану через `iptables -L OUTPUT` без sudo — non-root `node` user (наш навмисний least-privilege sudoers, лише `init-firewall.sh` passwordless) не має прав це прочитати, тому тест завжди SKIP навіть коли firewall реально активний. Прибрано перевірку, лишились самі curl-тести (не потребують root).
  4. `Dockerfile`: `FROM node:20`, а актуальний `@anthropic-ai/claude-code` вимагає Node ≥22 (EBADENGINE warning при білді). Піднято на `node:22`.
  Після фіксів: `make verify-devcontainer`, `verify-whitelist`, `verify-sandbox`, `verify-firewall` — усі 4 пройшли реально всередині запущеного devcontainer (Docker Desktop, WSL2-бекенд).

## 2026-07-17 (M3 capstone вже здано; 3 нові баги знайдено post-submission при реальному відкритті devcontainer)
- **3 нові баги в devcontainer знайдено й виправлено через реальний rebuild + login-тест** (виявлено вже після merge в `master` і після здачі capstone — попередній runtime-тест 2026-07-16 їх не ловив):
  1. `init-firewall.sh` мав CRLF на диску (Windows checkout з `core.autocrlf=true`, без `.gitattributes`) — ламало shebang (`#!/bin/bash\r`), `postStartCommand` падав з misleading "No such file or directory" (виглядало як "файла нема", хоча `COPY` в Dockerfile відпрацював коректно — сам git-blob був LF). Баг пролежав непоміченим, бо VS Code довго перевикористовував (`docker start`) старий контейнер без rebuild образу. Фікс: `.gitattributes` (`eol=lf` для `*.sh`/`Dockerfile`).
  2. Логін Claude Code: `ALLOWED_DOMAINS` мав `api.anthropic.com`, але не `console.anthropic.com`/`claude.ai` — OAuth-флоу логіну мовчки блокувався default-deny фаєрволом. Фікс: додано обидва домени в `init-firewall.sh` і синхронно в `.claude/settings.json` (звірка — `npm run verify`).
  3. Named volumes (`claude-code-config-*`, `claude-code-bashhistory-*`) належали `root:root` — Docker створює точку монтування volume від root незалежно від `USER node` у Dockerfile. `node` не міг писати токен логіну в `$CLAUDE_CONFIG_DIR` → "Failed to retrieve auth status after login" навіть після фіксу (2). Фікс: `chown -R node:node` додано в `init-firewall.sh` (виконується від root через уже дозволений sudo, без розширення sudoers).
  Після фіксів: rebuild + логін Claude Code реально пройшов у контейнері (перевірено навмисним відкатом прав на `root` перед rebuild, щоб підтвердити самозцілення саме через новий код скрипта, а не залишковий ручний фікс).

## 2026-07-18 (задача 2: Next.js scaffold + Vercel деплой)
- **Next.js вручну заскафолджений, не через `create-next-app`.** Чому: `create-next-app` відмовляється ініціалізуватись у непорожній директорії, а корінь репо вже містить `docs/`, `scripts/`, `tests/`, `.claude/` тощо поза його вбудованим allowlist безпечних файлів. `tsconfig.json` звужено до `include: app/**` і виключає `docs/` — інакше `tsc` підхоплював чужий express-стартер курсу з `docs/course/.../3.9-starters/nodejs-typescript`, у якого немає своїх `node_modules`, і білд падав.
- **Vercel: Git-інтеграція (import repo), НЕ CLI + токен.** Чому: автодеплой на push/merge без потреби передавати/зберігати `VERCEL_TOKEN` у цій сесії; це не перетинається з CI/CD Module 9 (там — PR-флоу/авто-рев'ю, тут — суто continuous deployment самого Vercel).
- **Firewall whitelist домену мусить бути точним hostname, не apex-доменом.** Чому: `init-firewall.sh` резолвить кожен домен у список IP через DNS (`getent`) і матчить по IP/порту, не по SNI/hostname. Перевірено: apex `vercel.app` резолвиться на іншу IP (216.198.79.3), ніж реальний проєктний субдомен `tax-navigator-red.vercel.app` (216.198.79.131) — не спільний anycast. Додано саме проєктний домен в обидва файли (`.claude/settings.json` + `init-firewall.sh`), `npm run verify` звіряє відповідність.
- **Self-modification `.claude/settings.json` (permissions/sandbox) заблоковано на рівні Claude Code класифікатора, навіть з явним дозволом Mike в чаті.** Процедурна нотатка: такі правки (allowlist доменів, permissions) Mike вносить руками сам; Claude Code може редагувати лише похідні файли (напр. `init-firewall.sh`), не сам `.claude/settings.json`.

## 2026-07-24 (анкета-навігатор: середовище і дизайн результату)
- **`/workspace` лежить на 9p/drvfs (диск Windows) — це і була причина «зависань» `next dev`, `next build` і jsdom-воркера vitest.** Не seccomp, як я записав спершу: вимір дав 9.63 мс на `stat` проти 0.01 мс на ФС контейнера, тобто ~500×. `require("jsdom")` не висне, а тягне 3615 модулів по ~22 мс = 80 с, і vitest убиває воркер по 60-секундному таймауту. Контрольний прогін на копії в `/tmp`: `next dev` — Ready за 2.3 с, `npm run test:ui` — 10 тестів за 5.8 с. Рішення: перенести репо на ext4 всередині WSL2; том для `node_modules` — запасний варіант, бо `inotify` через 9p не працює і HMR довелось би тримати на polling.
- **Збережений крок анкети — стеля, а не істина (`resumeIndex`).** Виручка навмисне не переживає F5, тож сліпе відновлення кроку висадило б людину після порожнього поля і вона дійшла б до результату з діркою у відповідях. Повертаємось на збережений крок АБО на перший незаповнений — дивлячись що раніше.
- **Ввід виручки стає слайдером по інтервалах 2 500 zł, а не точним полем.** Дві причини, крім приватності. (1) Один масштаб на всьому шляху: досі точні 15 000 при шерингу перетворювались на 12 500, і отримувач лінка бачив інші числа, ніж автор; коли інтервал є єдиним джерелом, ввід/сховище/лінк/розрахунок збігаються завжди. (2) Пороги zdrowotnej — 60k і 300k на рік, тобто 5 000 і 25 000 на місяць; при кроці 2 500 обидва є межами інтервалу, тож жоден інтервал не сідає верхи на поріг. Нинішні 5 смуг грубші: «25k+» рахувалось від 32 000 і для 26k, і для 60k.
- **Сторінка результату: порівняльна таблиця + акордеони, не стрічка карток.** Числа сценаріїв читаються порівнянням між собою, тож вони мають бути в одному полі зору; ризик, джерела й пояснення розкриваються по кліку, а не займають екран за замовчуванням.
- **Hooks парсять stdin через `node`, а не `jq`; сам `hooks.suggested.md` — перехідний файл.** Чому node: `jq` у базовому образі не було, тож приклади хуків із документації Claude Code мовчки не спрацьовували — хук «виконувався» і нічого не робив, що гірше за явну помилку (`jq` тепер доустановлено, але переписувати назад немає сенсу). Чому перехідний: хуки стають робочими лише в `.claude/settings.json`, а той у git — після вставки блоку ті самі команди житимуть у двох місцях і розійдуться. Свідомо НЕ додано два хуки: прогін тестів після кожної правки (~7 с на кожен Edit — роздратування без користі, бо тести й так ганяються перед комітом) і блокування коміту на «Mike не подивився UI» (хук не може знати, дивився чи ні — це лишається правилом, не автоматикою).
